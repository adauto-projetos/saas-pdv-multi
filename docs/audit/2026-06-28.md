# Auditoria Técnica — 2026-06-28

## TL;DR

Auditoria técnica completa do SAAS PDV.multi (Next.js 16 + React 19 + Drizzle + PostgreSQL com RLS, monolito fullstack). **Resultado geral: 8.9/10 — Healthy.** O projeto está sólido para um MVP de fundador solo: o isolamento entre lojas (multi-tenancy) é server-derived e protegido por duas camadas (filtro `tenant_id` na aplicação **mais** Row Level Security no Postgres), o dinheiro é tratado corretamente como inteiro em centavos, e nenhum segredo está commitado no git. **Nenhum problema Critical.** As três prioridades reais são preventivas: (1) o `SESSION_SECRET` cai silenciosamente num valor padrão público se a variável não estiver setada em produção — risco latente de falsificação de sessão; (2) `drizzle-kit push` apaga as policies de RLS e hoje depende de disciplina do operador para reaplicar — risco de vazamento entre lojas; (3) uma dependência (`undici`) tem uma CVE High com correção não-quebrante disponível. Tudo isso se resolve com ajustes pequenos e localizados.

## Recommendations

Ações priorizadas por impacto de negócio. Owner = Founder/Dev (projeto solo). Effort: S (≤1h) · M (≤meio dia) · L (≥1 dia).

- [ ] **[P1] Fail-fast no `SESSION_SECRET` em produção** — lançar erro no startup se `SESSION_SECRET` ausente/curto quando `NODE_ENV==="production"`; manter o default só em dev (`lib/auth/session.ts:14`) — Founder/Dev — S
- [ ] **[P1] Garantir RLS sempre reaplicado no deploy** — fazer o caminho de deploy/prod sempre rodar `db:rls` após qualquer push (comando único), e adicionar uma assertion em CI/startup que consulta `pg_policies` e falha se alguma tabela de negócio estiver sem `tenant_isolation` — Founder/Dev — M
- [ ] **[P1] `npm audit fix`** — resolve a advisory High de `undici` (não-quebrante) — Founder/Dev — S
- [ ] **[P2] Resolver a história de migrations** — decidir push-only (documentar, aposentar `db:migrate`, tratar `meta/0000_snapshot.json` como fonte) OU regerar migrations com `drizzle-kit generate` e adotar `db:migrate` — Founder/Dev — M
- [ ] **[P2] Mover persistência do super-admin para um service/repository** — tirar `db.transaction`/`update`/`select` de `actions.ts` para `lib/services/subscriptions/` (repository já existe); a action só valida, chama `requireFounder()`, invoca o service e `revalidatePath` — Founder/Dev — M
- [ ] **[P2] Atualizar CLAUDE.md** — reescrever "Status" e "Implementation Patterns" para refletir o estado atual (MVP+ com 17 features, v0.10.0) e rodar `/add.xray` para gerar a skill `project-patterns` — Founder/Dev — S
- [ ] **[P3] Adicionar índice em `override_log`** — `index("override_log_tenant_created_idx").on(tenant_id, created_at)` (`db/schema/override-log.ts`) — Founder/Dev — S
- [ ] **[P3] Batch do N+1 em `listOperators`** — trocar a query por operador por um único `IN (userIds)` ou join (`lib/services/users/operator-service.ts:70-87`) — Founder/Dev — S
- [ ] **[P3] Teste de regressão de tenant filter** — assertion de que toda query owner-connection em `operator-data`/`permission-data`/`override-data`/`audit-data` inclui predicado `tenant_id`; avaliar RLS scoped/SECURITY DEFINER para gerenciamento de membros — Founder/Dev — M
- [ ] **[P3] Centralizar Zod inline** — mover `loginSchema`/`signUpSchema`/`receiptSchema` para `lib/validation/` — Founder/Dev — S
- [ ] **[P4] Documentar vars de prod no `.env.example`** — bloco comentado com `POSTGRES_PASSWORD` e nota de que `SESSION_SECRET`/`R2_*` são obrigatórias em prod — Founder/Dev — S
- [ ] **[P4] Padronizar nome de arquivo de componente** — escolher PascalCase (de-facto maioria) e atualizar CLAUDE.md, ou renomear para kebab-case; alinhar a pasta `admin/` — Founder/Dev — M
- [ ] **[P4] Limpezas cosméticas** — corrigir comentário "Supabase" em `db/index.ts:6-9`; remover `components/PDVApp.jsx`/`.css` se não usados; confirmar gap de ID 0012; resolver porta 80 prod vs proxy — Founder/Dev — S
