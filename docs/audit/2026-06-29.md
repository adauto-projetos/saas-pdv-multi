# Auditoria Técnica — 2026-06-29

## TL;DR

Auditoria técnica completa do SAAS PDV.multi (Next.js 16 + React 19 + Drizzle + PostgreSQL com RLS, monólito fullstack). **Resultado geral: 8.9/10 — Healthy**, estável em relação à auditoria de 2026-06-28 ({{doc:AUDIT-2026-06-28}}). **Nenhum problema Critical e nenhum High de segurança.** A notícia mais importante é que as três prioridades P1 da auditoria anterior foram **verificadas como corrigidas**: (1) o `SESSION_SECRET` agora faz *fail-fast* em produção (`lib/auth/session.ts:21-29` + guard de boot em `scripts/verify-prod.ts:27-36`); (2) o footgun do RLS — `drizzle-kit push` derruba as policies — ganhou uma rede de segurança que **aborta o container** no boot se qualquer tabela com `tenant_id` ficar sem RLS (`verify-prod.ts:46-61`); e (3) a camada super-admin foi movida para services, com até um teste de regressão que proíbe `drizzle-orm` na action (`actions-no-drizzle.test.ts:28-29`). Os índices e o N+1 apontados antes também foram resolvidos. **As cinco questões High de hoje são todas "higiene", não falhas de segurança:** 2 de arquitetura (acesso Drizzle cru na conexão *owner* dentro de arquivos de layout/UI — `app/(app)/layout.tsx`, `caixa/receipt-actions.ts`), 2 de documentação (README ainda é o boilerplate do create-next-app, e a linha "Status" do CLAUDE.md está desatualizada em versão e contagem de domínios) e 1 de infra (o `scripts/deploy.sh` versiona IP de produção, usuário `root` e caminho da chave SSH). A única ressalva de segurança que persiste é o SEC-001: alguns módulos (auditoria, operadores, permissões, admin) leem na conexão *owner* que **bypassa a RLS**, apoiados só no filtro `tenant_id` da aplicação — correto hoje, mas sem a "última linha de defesa" do banco.

## Recommendations

Ações priorizadas por impacto de negócio. Owner = Founder/Dev (projeto solo). Effort: S (≤1h) · M (≤meio dia) · L (≥1 dia).

- [ ] **[P1] Tirar credenciais/host de deploy do Git** — mover `SERVER`/`KEY` de `scripts/deploy.sh:8-9` para um `.env.deploy` ignorado; usar usuário de deploy não-root; guardar a chave fora de `/tmp` com permissão `600`; trocar `StrictHostKeyChecking=no` por `accept-new` (INF-001) — Founder/Dev — S
- [ ] **[P1] Atualizar README.md e a linha Status do CLAUDE.md** — substituir o boilerplate do create-next-app pelo fluxo real (`docker compose up -d` → `.env.local` → `npm run db:setup` → `npm run dev`) ou apontar pro CLAUDE.md; corrigir Status para v0.13.0 / ~15 domínios / features até 0022C; idealmente ler a versão do `package.json` em vez de hardcodar (DOC-001/002) — Founder/Dev — S
- [ ] **[P1] Reduzir a superfície do bypass de RLS** — cobrir cada método das data layers owner-db (audit/operator/permissions/admin) com teste de isolamento por tenant; avaliar uma policy de leitura por-tenant em `tenant_members` que permita migrar essas leituras para `withUserRls`, eliminando o bypass (SEC-001) — Founder/Dev — M
- [ ] **[P2] Eliminar o Drizzle cru na camada UI/action** — extrair o read de recibo para `lib/services/sales/data.ts` (ARCH-001); rotear os dados dos layouts `(app)`/`(admin)` por um service sob `withUserRls`, reservando a conexão *owner* só para login/onboarding/seed/DDL (ARCH-002/003); estender o teste `actions-no-drizzle` para `app/**` — Founder/Dev — M
- [ ] **[P2] Rate-limit no login** — limitar tentativas por IP+email com backoff (contador em DB/Redis ou header do proxy), idealmente captcha após N falhas (SEC-003) — Founder/Dev — M
- [ ] **[P2] Adicionar CI mínimo** — workflow GitHub Actions rodando os 4 gates (`lint`/`typecheck`/`test`/`build`) em cada PR; opcionalmente subir Postgres de serviço para os testes de RLS (INF-003) — Founder/Dev — M
- [ ] **[P3] Indexar as FKs mais quentes** — criar índice em `sale_items.product_id` e `comanda_items.product_id` (elimina full-scan no delete de produto); avaliar as FKs de autoria conforme o histórico cresce (DATA-001) — Founder/Dev — S
- [ ] **[P3] Documentar as env vars faltantes** — adicionar `FOUNDER_EMAIL` e `PRINTER_DEVICE` ao `.env.example` (comentadas/opcionais, explicando quando são necessárias) (INF-002) — Founder/Dev — S
- [ ] **[P3] `middleware.ts` de borda** — exigir o cookie de sessão em `(app)`/`(admin)`/`api` como rede de segurança, sem remover os guards de action (SEC-002) — Founder/Dev — M
- [ ] **[P3] Mover os `select` crus de `print-service.ts` para `print-data.ts`** — fechar o último domínio fora do padrão service/data (ARCH-004) — Founder/Dev — S
- [ ] **[P4] Higiene de schema/migration** — remover `db/migrations/meta/` (ou isolar os `*_rls.sql` num diretório próprio + ajustar `apply-rls.ts`/`out`); renomear `subscriptions.ts` → `subscription-log.ts`; comentar `0008_subscription_lifecycle.sql` como não-executado (DATA-002/003/004) — Founder/Dev — S
- [ ] **[P4] `npm audit fix`** — resolver as 6 moderate (dev/build); acompanhar bumps de `drizzle-kit`/`next` (INF-006) — Founder/Dev — S
- [ ] **[P4] Confirmar e documentar a terminação TLS** e a estratégia de schema em prod; garantir backup do volume `pdv_pgdata` antes de deploys que alterem schema; alinhar variante da imagem Postgres dev/prod (INF-004/005/007) — Founder/Dev — S
